Structured Context Windows: A Composable Memory Architecture for Agentic Systems

Structured Context Windows: A Memory-Oriented Orchestration Layer for LLM and Multi-Modal Agents
Abstract— We introduce Structured Context Windows (SCWs), a general-purpose memory structuring primitive for agentic AI systems. An SCW is to a language model’s context window what a chat session is to an entire project: a containerized workspace that persists and manages information beyond a single prompt, enabling composable, testable, and observable units of reasoning. SCWs extend across data modalities and serve as an orchestration layer that unifies short-term scratchpads, episodic memories, and long-term knowledge stores. We describe the SCW object model and its strict invariants, a four-tier memory architecture (scratchpad, episodic, persistent, permanent) with reflex-based gating, and a control plane that governs policies, anchors, and observability separate from the data-plane of model calls and tool invocations. Implemented within our Maxey0 agent framework, SCWs provide dynamic spawning of sub-agents, deterministic memory promotion, and inter-agent communication channels – all aiming to improve reliability and scalability of large language model (LLM) and multi-modal pipelines. We report deployment case studies using GPT-4 (OpenAI ChatGPT) and Claude 2 (Anthropic “Sonnet 4.0”), demonstrating that SCWs can encapsulate complex multi-step tasks and multi-agent coordination within today’s context length limits. The SCW approach offers a new avenue toward containerized cognition, where each unit of work in an AI system carries its own context, policy, and memory scope – enabling safer and more transparent agent operations.
1. Introduction
Large language models have a fixed context window for prompts, traditionally treated as a transient scratchpad that vanishes once filled. This limitation complicates long-horizon reasoning and multi-step tool usage, especially in multi-agent or multi-modal scenarios. Current approaches to extend an LLM’s effective memory include appending lengthy conversation histories, using external vector databases for retrieval, or fine-tuning larger context models. Yet these tactics often lead to cluttered or ungoverned context, where relevant information competes with stale data and hallucinations go unchecked. Open-source agent frameworks (e.g. LangChain, AutoGPT) have emerged to bolt on memory via databases and summary buffers, but they lack a formal unit of context management – there is no mechanism for structured workspaces with lifecycle and policy control. We propose Structured Context Windows (SCWs) as a principled solution. An SCW is a bounded execution environment inside or alongside an LLM’s context window, complete with its own rules, tools, persona, and memory scope. In effect, SCWs turn the ad-hoc context into a first-class object: a context capsule that can be spawned, audited, and dissolved independent of the surrounding chat. This is analogous to how modern IDEs or operating systems use isolated processes or containers – here we isolate cognitive “threads.” Just as an entire chat session provides continuity for a project (persisting relevant info across user queries), an SCW provides continuity within a complex reasoning task across multiple model calls. SCWs are composable (they can chain or nest within a larger task graph), testable via deterministic replays, observable via built-in telemetry, and policy-scoped to enforce domain-specific constraints. Crucially, SCWs are designed to work across all data types: while our examples focus on text-based LLM agents, the SCW abstraction applies equally to multi-modal agents handling images, audio, or sensor data streams – any context that benefits from structured memory management. In the following, we formalize the SCW concept and describe its implementation in our Maxey0 agentic framework. We present the SCW object model and invariants that guarantee no information escapes reflexive vetting (mitigating hallucinations). We then detail the 4-tier memory architecture that underpins SCWs, organizing memory into scratchpad, episodic, persistent, and permanent layers with deterministic promotion policies. We explain how SCWs operate with a clear control plane vs data plane separation, how they spawn and dissolve based on triggers, and how they facilitate agent-to-agent (A2A) communication through sealed “tunnels.” Finally, we provide case studies of SCWs in action with GPT-4 and Claude, illustrating improvements in continuity and reliability, and discuss how SCWs align with emerging research on agent memory and multi-agent orchestration.
2. SCW Object Model and Architecture
Definition: An SCW is a fundamental execution container for reasoning. It encapsulates a portion of the LLM’s computation along with associated state. Formally, an SCW is an object with: (i) a unique ID (SCW-<timestamp>-<index>), (ii) input shards (the pieces of context or data it takes in), (iii) a set of policies governing its behavior, (iv) output shards (results it produces), and (v) observability hooks for logging and metrics. Shards here denote minimal units of information (e.g. a thought, a tool result, a citation) that flow through the SCW. SCWs are not mere metaphors; they have strict invariants. For example, no shard can bypass the reflex gating (see §3) within its SCW, every output carries a lineage record (tracing which SCW and policy produced it), SCWs have a finite lifespan (they dissolve after completing their task or timing out), and all reflex operations are deterministic under fixed conditions. These invariants ensure that SCWs behave predictably and can be audited or replayed exactly. In essence, each SCW is a self-contained workspace with a defined start and end, and a memory of its own that can be promoted or discarded per rules. Control Plane vs. Data Plane: Borrowing from networking and OS design, SCWs maintain a hard separation between control logic and data handling. The control plane includes meta-level governance: the SCW’s policies (e.g. domain rules, safe-completion guidelines), role-based access controls (RBAC) for tools or APIs, scheduling parameters, and resource quotas. Control plane data might include trusted anchors – factual references or constraints injected to guide the SCW’s operation (for instance, a company knowledge base document that the SCW must treat as ground truth). This plane is immutable from the model’s perspective; the LLM cannot alter its own policies or lift its quotas. The data plane is the content being processed: the natural language tokens, embeddings, intermediate computations, and tool outputs that constitute the “work” of the SCW. By isolating governance from generation, SCWs prevent prompt-level content from manipulating the rules – a vital property for alignment. For example, an LLM might generate a query that, if taken naively, calls an unsafe tool, but the SCW’s control plane can block this via policy before execution. This design echoes the principle of least privilege: the generative model operates within bounds set by the SCW’s container. SCW Stack and Multi-Modal Extensibility: Conceptually, an SCW carries its own stack of functional modules. As shown in Figure 1, an SCW definition may specify (a) a persona or role – e.g. a “Python Assistant” SCW might emulate a coding agent with a specific tone; (b) a set of tools it can use – e.g. web search, calculators, or image APIs; (c) a memory scope – which of the four memory tiers (scratchpad, episodic, persistent, none) are active; (d) behavioral constraints or rules – e.g. “avoid first-person pronouns”, “cite sources for claims”; and (e) state tracking instrumentation – markers or counters that track its progress. All these aspects are encapsulated within the SCW and apply only to it. This means different SCWs in the same overall session can have entirely different behaviors and knowledge – a form of contextual encapsulation impossible in a monolithic prompt. For instance, one SCW could be a vision module that only processes images with a vision model, another a text analyst; they communicate via structured outputs rather than sharing an unstructured prompt. SCWs thus generalize beyond text: any modality or subsystem can be wrapped in an SCW interface, so long as it abides by the memory and reflex rules. This enables multi-modal agentic systems to coordinate by passing verified results between SCWs (see §4.3 on tunnels) instead of raw data in prompts. Figure 1: SCW Architecture Diagram. Each SCW is a container with layers for rules, memory, tools, persona, state tracking, etc. The SCW’s memory slice can be configured per task – including no memory for stateless operations, or specific combinations like a scratchpad + persistent link. All SCWs share a common runtime interface, allowing the orchestrator to spin up or shut down SCWs on demand with predefined configurations. Below is an illustration of an SCW’s internal structure:

3. Memory Tiering and Reflexive Memory Governance
A core motivation for SCWs is to structure the memory of an agentic system. Rather than a monolithic context or an unbounded knowledge base, Maxey0 uses a four-tier memory model called memre, which every SCW interacts with. The tiers are: (1) Scratchpad, (2) Episodic, (3) Persistent, and (4) Permanent memory. They differ in longevity, scope, and governance, as described below and summarized in Table 1.
Scratchpad (Tier-1): an SCW’s local working memory. This is ultra-ephemeral, with a time-to-live often just minutes. It exists only for the SCW’s lifespan and is used for token-level reasoning, hypothesis generation, and intermediate calculations. In implementation, scratchpad data lives inside the model’s prompt or a temporary store (e.g. a Python dict in memory). Once the SCW dissolves, its scratchpad is wiped unless explicitly saved upward. The scratchpad allows the agent to “think” freely with no long-term consequence – analogous to scribbling on a notepad that will be discarded.
Episodic (Tier-2): a short-term memory spanning a single task or a coherent session (e.g. one question-answering job, or one run of a tool chain). Episodic memory logs what happened inside the SCW – including intermediate steps, model outputs, tool calls, and reflex decisions. This can be stored in a durable but transient medium such as a Redis log or an in-memory database. Episodic memory lasts at least for the duration of a pipeline run (which may involve multiple SCWs chained) and is primarily used for auditing and potential recovery. If an SCW fails or needs to be replayed, the episodic log provides the step-by-step transcript. However, episodic data is not automatically available to other tasks unless promoted; it’s scoped to the run. Think of it like an application log or an agent’s short-term memory of what it just did.
Persistent (Tier-3): a long-lived, cross-task memory store for verified knowledge. This is typically implemented as a graph database (Neo4j) or a vector index, or a combination, collectively forming what we call the knowledge graph. Only information that has passed certain quality gates (via reflex promotion, described shortly) enters persistent memory. Examples include facts found to be true, conclusions of analyses, or important decisions. In our architecture, persistent memory is represented as nodes and relationships in Neo4j (for symbolic facts and provenance) and as vector embeddings (for semantic search). Persistent memory accumulates across runs and even across agent instances – it’s the agent’s growing knowledge base. Crucially, SCWs cannot write directly to persistent store except through the reflex mechanism; this prevents raw hallucinations from polluting long-term memory. Each committed entry carries lineage metadata linking it back to the originating SCW and the policy that allowed it. Persistent memory provides organizational knowledge sharing: multiple agents (or multiple SCWs in future tasks) can retrieve these vetted facts via queries, enabling cumulative learning.
Permanent (Tier-4): an immutable archive for only the most trusted and crucial information. This layer is akin to an append-only ledger or a model’s parameter memory – once written, entries are fixed. We reserve permanent memory for compliance-critical or “ground truth” facts that should never be revised or removed (except by human intervention). In practice, this might be a secure ledger or a blockchain log of key decisions, or a frozen knowledge graph schema. The Permanent tier is rarely populated; it’s meant for things like finalized legal rules, confirmed scientific axioms, or output from model training (e.g. distilled knowledge that’s used to inform model weights). Entries here often require multi-agent consensus and even human sign-off (see below) to reach. You can think of it as the write-once, read-many (WORM) memory with cryptographic sealing.

For example, once an SCW completes its task, it might promote key results from scratchpad to episodic (so the next SCW or a supervisor agent can see them), or even to persistent if they are deemed verified facts. Reflex Mechanism (Hallucination Control): At the heart of SCW memory governance is the reflex, a built-in introspection and self-correction loop that every SCW executes before committing memory. The reflex mechanism classifies every output shard into one of three states: REAL, WRONG, or HALLUCINATION. These roughly correspond to supported, contradicted, or unsupported content. To perform this classification, we use multiple signals: stability (S) – how consistent the shard is under perturbations or re-phrasing; completeness (C) – whether the shard contains sufficient context/evidence or is just a fragment; and trust (T) – how well the shard aligns with known ground truths or anchors. For example, a claim that is repeated with variation (low entropy) in multiple prompts and matches a known fact would score high on all three and be classified REAL, whereas a one-off claim not found in any references with contradictory context would be WRONG or HALLUCINATION. Reflex policies then take action based on this classification and the tier boundaries described above. SCWs embed four deterministic reflex operators: Kill, Quarantine, Promote, and Tunnel.
Kill: immediately discard shards flagged as hallucinations or otherwise invalid. This prevents them from influencing any memory tier. For instance, if an SCW (say an LLM agent) makes up a false citation, the reflex would mark it as hallucination and the Kill operator would ensure it’s not written to episodic log except perhaps as part of an error trace.
Quarantine: retain shards in a quarantined state for inspection but block their promotion. Unstable or ambiguous outputs often fall here. They remain accessible in the SCW’s own context (for potential self-correction or review), but will not propagate to persistent memory unless resolved. Quarantining is like isolating suspicious code in a sandbox – it’s there but inert from the system’s perspective.
Promote: as discussed, this operator moves shards upward when thresholds are met. It is invoked after classification confirms a shard is REAL (or at least not deemed a hallucination by policy). Promotion may target different tiers: e.g. promote from scratchpad to episodic (making it visible to other SCWs in the same pipeline), or episodic to persistent (making it part of long-term knowledge). Promotion events are the primary means by which lasting learning happens in the system.
Tunnel: package a shard (typically a verified micro-fact or outcome) for transfer to another SCW. Tunneling is the mechanism for A2A communication: one SCW can send a distilled piece of information to another without exposing the entire context. For instance, after an SCW finishes a subtask, it might tunnel the result to a parent SCW or a coordinator agent. The “tunnel capsule” includes the fact content, evidence references, provenance (which SCW and policy produced it), and optionally a Time-To-Live (TTL) or access permissions. By design, only content that passes reflex checks can be tunneled. If a shard is quarantined or unverified, the tunnel operator will refuse it, similar to how a network firewall drops unauthorized packets. This ensures that cross-agent communication remains factual and bounded – agents communicate findings, not speculations.
These reflex operators run deterministically and are versioned as part of the SCW definition. Determinism means that given the same input and policy, the reflex will always classify and act the same way (important for reproducibility and debugging). Versioning means we can update the policies (e.g. tighten thresholds) and know which SCWs used which policy version, aiding traceability. In effect, SCWs implement a form of internal guardrail for LLMs: rather than relying purely on external human feedback or fine-tuning to reduce hallucinations, we enforce a structured self-check before an agent’s output is trusted or remembered. This draws inspiration from recent frameworks like Reflexion (Shinn et al., 2023) which use iterative self-evaluation to correct LLM outputs, but goes further by making it an integral, containerized part of the agent’s run. Empirically, this multi-agent reflex approach resonates with findings by Park et al. (2023) and Gao et al. (2023): generative agents are more stable and factual when they incorporate memory and social feedback loops (i.e., multiple agents or modules cross-verifying each other). Lineage and Provenance Tracking: Every shard that survives within or across SCWs is logged in a provenance graph. We model this in Neo4j as a property graph where shards, SCWs, agents, and policies are nodes, and relationships denote generation, usage, and promotion. For example, when SCW-17 (node) generates a shard about “X is Y”, we create relationships like (:Shard)-[:GENERATED_IN]->(:SCW) and [:PROMOTED_BY]->(:Policy) linking that shard to SCW-17 and the policy node representing the reflex that allowed promotion. If that shard is later used by another agent, we might add [:OBSERVED_BY]->(:Agent) or if it’s tunneled [:TUNNELED_TO]->(:SCW-18). The point of this graph is to enable complete auditing and replay: one can query any fact in persistent memory and trace exactly how it was derived. This is invaluable in enterprise or safety-critical contexts – it provides a chain-of-custody for information, supporting compliance (e.g. knowing which AI system output which insight under what rules) and debugging (if a wrong fact got in, you can pinpoint which SCW/policy failed to catch it). The lineage graph also allows research on the agent’s knowledge evolution: e.g. detecting when two independent SCWs discovered the same fact (and possibly merging their evidence), or measuring how often shards get demoted after promotion (a sign of policy tuning need). Finally, the Permanent tier, when used, is integrated via special relationships like :ASSERTED_TRUE or cryptographic attestation subgraphs. Since Permanent memory is write-once, each entry might carry signatures from multiple agents or humans confirming it. SCWs enforce that no entry reaches Permanent without such consensus (often K-out-of-M voting among agents plus an operator sign-off). This approach draws on concepts from distributed consensus and sensor fusion – in fact, we dub one policy schema Drone Operating System Triangulation (DOT), inspired by how drones combine inputs from multiple sensors. In DOT, a single agent’s assertion is never sufficient; a claim must be corroborated by independent instruments (retrieval, calculation, etc.) and potentially human review. This multi-agent consensus model significantly reduces false positives entering long-term memory, echoing observations by Gao et al. (2023) that large-scale multi-agent simulations achieve better robustness with arbitration mechanisms.
4. SCW Lifecycle and Orchestration
SCWs are dynamic constructs. They are spawned, they perform work (often in cooperation with others), and they eventually dissolve, handing off results. This lifecycle is tightly orchestrated to maximize modularity and minimize interference. Here we outline how SCWs are created and managed, and how they communicate in a larger agent system. 4.1 Spawn Triggers: Instead of keeping one gigantic context window for everything (which risks context overflow and intermingled topics), the system spawns new SCWs when appropriate. Spawning can be triggered by several conditions:
Memory Pressure: If an ongoing SCW’s scratchpad is getting too large (e.g. > N tokens nearing context limit), a new SCW may spawn to offload some information to persistent storage. This could be called a “Memory Persistence SCW” – it writes summary or key data to the knowledge graph and then dissolves, relieving the pressure.
Task Complexity: If the problem at hand scores high on a complexity heuristic (say, requiring diverse subtasks or exceeding a reasoning entropy threshold), the system spawns specialized SCWs for sub-tasks. For example, encountering a complex math problem might spawn an “Analytical SCW” with a math toolset, or a coding question might spawn a “Python SCW”. This is akin to an expert system dynamically assembling a team of specialist agents.
Agent Coordination: When multiple agents (or SCWs) need to collaborate (e.g. more than 3 agents required for a sub-problem), a collaborative SCW is spawned to serve as a shared workspace. This SCW would have a memory scope that allows it to read specific episodic logs from each contributing agent, and its role is to merge or reconcile their inputs. After finishing, it might promote a consensus result to persistent memory.
Context Overflow: If the current conversation or workflow is changing topic or simply running long, the system can spawn a fresh SCW to carry forward only the essentials. For instance, after a long discussion, SCW-Alpha can summarize its state and then spawn SCW-Beta with that summary as initial input (and maybe a link back to Alpha’s persistent entries for reference), then dissolve Alpha. This is similar to how human teams might end one meeting and start a new one with a summary of the last – it prevents irrelevant backlog from clogging the active context.
Episodic Boundary: When a significant milestone or “insight” is reached (detected via an importance score crossing a threshold), a Memory Bridge SCW may spawn. Its job is specifically to take the episodic memory (short-term log) of that session and encode it into the persistent knowledge graph. This can happen at the end of a task (ensuring the session’s learnings aren’t lost) or even mid-way if something worth preserving emerges. The Memory Bridge SCW operates almost like a background save thread, archiving knowledge.
Spawning an SCW involves allocating it an ID, loading its definition (which tools, persona, memory access it has), and seeding it with an initial prompt or input shards. In Maxey0’s implementation, a central orchestrator service manages this process. The orchestrator monitors the conversation or task graph and, upon a trigger, instantiates the new SCW (which could run in a separate thread or even on a separate machine, depending on scaling needs). We use YAML-based configs to define SCW templates, so the orchestrator knows what type of SCW to spawn for a given trigger (e.g., “complexity > 0.8 → spawn SCW type Analytical with Tools=X, Role=Y”). 4.2 Execution and Coordination: Once spawned, an SCW enters the Work phase. It will consume its input shards, possibly call the LLM for some reasoning, invoke tools, accumulate new shards in its scratchpad, and so on. In isolation, this looks like a standard prompt-response or chain-of-thought sequence, but the key is that it’s happening inside the SCW’s sandbox. SCWs can run synchronously or asynchronously relative to others. If SCW-B is spawned by SCW-A as a subtask, A might wait (synchronous) for B to finish and return a result (perhaps via a tunneled shard) before A continues. Alternatively, for independent tasks, SCWs run in parallel (asynchronous), with no waiting – the orchestrator just collects their outputs when ready. We represent dependencies as a directed acyclic graph (DAG) of SCW tasks. For example, “SCW-12 must complete before SCW-13 can start” is recorded as a relation SCW12 -> SCW13. In our Neo4j memory, we even log this as (:SCW {id:12})-[:DEPENDS_ON]->(:SCW {id:13}), giving a persistent record of the execution plan. This allows re-running or analyzing complex pipelines after the fact. Maxey0’s orchestrator supports flexible scheduling via YAML, e.g.:
pipeline:
  - SCW11: start
  - SCW12: after SCW11 +5s   # spawn SCW12 five seconds after SCW11 starts
  - SCW13: after SCW12       # wait for 12
  - SCW14: async             # spawn in parallel, no waiting
In this example, SCW11 runs first, SCW12 starts shortly after 11 (perhaps overlapping), SCW13 only starts when 12 finishes, and SCW14 runs independently (it might be an observer or logger). The orchestrator ensures each SCW logs its completion, and downstream dependencies are triggered accordingly. By structuring agent workflows as DAGs of SCWs, we impose order and modularity on what could otherwise be an unpredictable chain-of-thought. It also means failures can be isolated: if SCW12 crashes or goes awry, SCW13 might never start, and the orchestrator can decide how to handle that (maybe spawn a recovery SCW or alert a human). 4.3 Dissolution and Post-Mortem: When an SCW has finished its work (either by reaching a goal or being instructed to stop), it enters the Dissolve phase. The SCW runs its final reflex pass, promoting any remaining worthy shards and killing the rest, then shuts down. “Shutting down” includes writing a lineage record (in Neo4j, we tag the SCW node as completed with a timestamp and link it to a :PipelineRun entity). Essentially, the SCW becomes an immutable object in memory – it can be replayed but not altered. Dissolution frees up resources (e.g., context window budget, compute threads) and cleans the scratchpad. Notably, SCWs that are dissolved can be later reinstantiated or inspected. We can load the SCW’s inputs and outputs to replay its behavior exactly, or examine its logs in a dashboard. This supports post-mortem analysis: if something went wrong, developers or automated auditors can dig into that SCW’s trace without confusion from other concurrent tasks. SCW isolation limits any “blast radius” – an error or hallucination in SCW12 stays within it (perhaps quarantined) and does not automatically taint SCW13 or the persistent memory. Only if SCW12’s outputs passed reflex promotion would they become visible to others, which provides a controlled containment of faults (analogous to circuit breakers in distributed systems). Inter-SCW Communication: While isolated, SCWs are not silos – they often need to exchange information. We described the tunnel mechanism in §3, which is the primary means of passing concise, verified data. Implementation-wise, we realize A2A communication through a combination of Redis streams and direct graph writes. For immediate message passing (e.g., SCW12 needs to send a result to SCW11 which is waiting), a Redis pub/sub or stream channel labeled with the target SCW’s ID is used. The tunnel operator will publish the capsule to that channel. The receiving SCW (11) subscribes and upon receiving, it injects the information into its prompt or scratchpad as needed. For longer-term or less direct sharing, writing to the persistent store serves as communication: one SCW might persist a finding in Neo4j, and later another SCW can query the graph to retrieve relevant nodes. We found this dual approach useful – Redis for real-time, Neo4j for memory-based handoffs – to balance performance and durability. Any messages that could not be delivered (e.g. target SCW died) end up in a dead-letter queue for analysis, and metrics like A2A DLQ rate (dead-letter rate) are tracked. Scheduling and Parallelism: SCW orchestration introduces a question: how to efficiently manage potentially dozens of SCWs running concurrently. In our deployment, we treat SCWs as lightweight, mostly I/O-bound tasks (calls to LLM APIs or tools). We spawn them in a thread pool and even across containerized microservices. The Maxey0 orchestrator coordinates this, and a pool of SCW runner processes execute the SCWs in parallel when needed. This allows scaling to large agent swarms. For example, one experiment spun up ~120 SCWs in parallel to simulate an “expert committee” solving parts of a problem, akin to Gao et al.’s large-scale agent simulations. SCW scheduling policies ensure we don’t overload any single resource: we can cap how many SCWs use the LLM at once, or enforce that certain critical SCWs run serially for determinism. The ability to sequence and parallelize at the SCW level (instead of individual API calls) provides a more robust and observable scheduling unit. We log each SCW’s start and end, which agent invoked it, and any delays, enabling Gantt chart–like views of the agent’s overall execution timeline. Budgeted Printing: One practical feature we added is the budgeted printer – each SCW can enforce a word or token budget on its output. If an SCW is tasked with producing a report of max 500 words, the SCW’s printer will cut off or trigger a new SCW if the LLM tries to exceed that. This ensures compliance with expected lengths and prevents runaway generation. Overflow text is not lost: it’s captured as shards and can spawn a continuation SCW if needed (tiling strategy). This is particularly useful for structuring outputs that span multiple chunks or ensuring consistency in experiments (each SCW output is comparable in length). In our ChatGPT/Claude deployments, we indeed used this to split long analyses into multi-SCW sequences automatically.
5. Deployment Examples and Discussion
We integrated SCWs into two state-of-the-art LLM platforms to evaluate their practicality: OpenAI GPT-4 (via ChatGPT) and Anthropic Claude 2 (100k context, “Sonnet 4.0”). Our goal was to see if SCWs can operate within the constraints of current model APIs and whether they yield tangible benefits in complex tasks. Case Study 1: ChatGPT (GPT-4) as a Self-Orchestrating Research Assistant. We used GPT-4 (via the ChatGPT interface) to emulate an agent that writes a short research paper (in fact, an earlier draft of this very document). Without SCWs, GPT-4 is limited by its ~8k token context and a single-step chain-of-thought – it tends to either forget earlier instructions or mix content uncontrollably when the task is long. We therefore orchestrated the task into multiple SCWs inside the conversation. Specifically, we created SCWs for each major section of the paper: Introduction, Methodology, Results, etc., each with its own scratchpad. A top-level SCW (call it the “Editor SCW”) acted as a controller, deciding when to spawn a new section SCW. For example, once the Introduction SCW finished (and its content was validated via reflex to be on-topic and factual), the Editor SCW spawned a Background SCW, providing it a summary of the intro via a tunneled shard and some new references to cover. This proceeded through Conclusion, each SCW about ~500 tokens. ChatGPT as a platform does not natively support spawning concurrent threads, so we simulated it sequentially (the conversation kept context of the orchestrator’s decisions). Even so, the effect was clear: each section was far more coherent and self-contained, and we could enforce, say, that the Introduction SCW did not bleed into methodology details, because the Editor SCW only passed along certain info. The reflex mechanism caught a hallucinated citation in one section – the SCW quarantined that reference, preventing it from being included in the final persistent output. From the user’s perspective, this looked like the assistant systematically tackling the task section by section, always remembering key points (since they were promoted to persistent memory and then referenced by later sections) but not exceeding the scope. The final assembled paper was more consistent and had fewer logical jumps than an end-to-end generation. This demonstrates that even within a single ChatGPT session, thinking in terms of SCWs (with careful prompt engineering to mimic the spawn/dissolve) leads to better organized results. It also provides a blueprint for how one might implement SCW orchestration on top of API calls: by dividing a complex prompt into multiple API calls, each with its own context managed by an external program (our orchestrator script). Case Study 2: Claude “Sonnet” 100k – Long Conversations with Structured Memory. Anthropic’s Claude 2 model offers an extended context (up to 100k tokens), which we leveraged to test SCWs in a scenario of a sustained multi-agent discussion. We configured a scenario where a user asks a complex question (“Design a multi-modal AI system for real-time disaster response”), and the assistant spawns several SCWs internally: one to brainstorm requirements, one to research existing literature (using a tool via its API), one to propose an architecture, and one to critique the plan. Because Claude can take a very large single prompt, we actually injected all these SCW definitions and content into one conversation, effectively interleaving them (the SCWs communicated via the persistent memory within that single context). Claude’s 100k context became a hosting ground for multiple SCWs running interleaved. Each SCW was demarcated by system messages that enforced its rules/persona (Anthropic allows system-level instructions to some extent). The result was that Claude was able to handle a multi-faceted question by internally segmenting the work: e.g., the “Researcher SCW” pulled in factual data (we manually embedded some documents as part of its input), the “Planner SCW” was composing the strategy, and the “Critic SCW” was waiting to receive the plan and find flaws. This all happened in one continuous chat from the user’s view, but under the hood, the SCW structure ensured that, for instance, the Critic didn’t jump in before the plan was ready, and the Planner didn’t see the raw research documents until they were summarized by the Researcher SCW (to avoid overloading or poisoning its own generation). Claude’s large context made it possible to maintain the entire SCW network state in one prompt, something we can’t do with smaller contexts – but even Claude would have struggled if we naïvely concatenated everything. The SCW discipline meant each piece of context was only active where needed. This experiment also validated the multi-modal aspect: one SCW handled text (plans), another could have handled images or maps if we had an image analysis tool; by encapsulating different modalities’ processing in different SCWs and then merging results through persistent memory, we maintain clarity. The overall outcome was a thorough, multi-angle answer that was more grounded and balanced than a single agent response, achieved without human-interleaved prompting – the SCW orchestrator prompts did the coordination. Observability and Metrics: Throughout these deployments, we logged telemetry per SCW. Some key metrics: SCW success rate (how many SCWs achieved their goal versus timed out or errored), cross-SCW interference (how often one SCW’s context leaked into another’s – ideally near zero), reflex false positive/negative rates (did we ever mistakenly kill a good shard or let a bad shard through), and resource usage per SCW. We found high success rates (~93% in a 120-SCW build test) and very low interference (~5% cross-talk), indicating the isolation works as intended. The reflex thresholds occasionally needed tuning – initially a few true facts were quarantined because they lacked a second source, which we adjusted by lowering certain consensus requirements for low-stakes info. The budgeted printer ensured consistency in output sizes, which was useful for evaluation. In our tests, using SCWs did incur overhead (roughly 20% more total tokens used due to repeated context setup across SCWs), but the gains in result quality and manageability were significant for complex tasks. A qualitative benefit was explainability: after an SCW-based run, we could answer why the system gave a certain answer by tracing through the SCWs and their decisions. This is a step toward more transparent AI, aligning with calls for system interpretability.
6. Related Work and Positioning
Our Structured Context Windows approach intersects ideas from conversational memory in chatbots, agent architectures in multi-agent simulations, and software engineering principles like sandboxing and microservices. We briefly situate SCWs in context of prior art: Conversational Memory & Long Context: Efforts to extend LLM context (such as GPT-4 32k, Claude 100k) and external memory (e.g. Retro, RAG, vector databases) address the capacity issue but not the structural issue. They allow more data to be packed in, but as Park et al. (2023) note in their Generative Agents work, simply having a lot of memory is not enough – one needs to organize and retrieve it properly
arxiv.org
arxiv.org
. Their architecture uses natural language entries as memories and periodic reflection to summarize, analogous to our persistent memory and reflex promotion (they even had the agent pause to reflect every so often). SCWs generalize this by formalizing when and how such pauses or segmentations should happen (via spawn triggers and reflex checks). We view SCWs as complementary to large context windows: even if models in future have million-token contexts, segmenting the cognitive process into windows with local focus can prevent the model from entangling unrelated threads and reduce the risk of catastrophic forgetting or override of earlier info. Tool Use and Meta-Prompting: The ReAct framework (Yao et al., 2022) introduced interleaving reasoning and acting with tools in a single context. SCWs build on this concept by providing separate contexts for separate tools or subtasks, rather than interleaving everything in one. This avoids a common problem in ReAct-style agents: the prompt grows with each thought and tool result, and the model starts confusing different steps. Our approach is closer to the recently proposed Tree-of-Thoughts (Yao et al., 2023) which explores multiple solution branches: one could see SCWs as the tree nodes, each exploring a sub-problem branch which can be dropped or integrated later. However, SCWs add the critical notion of memory governance – Tree-of-Thoughts doesn’t explicitly handle what information should be retained or pruned beyond the tree search. Self-Reflection and Alignment: Several works have tried to mitigate LLM hallucinations and errors via self-reflection, as in Reflexion (Shinn et al., 2023) or Self-Critique techniques. These typically run the model in a loop: initial answer, critique, revised answer, etc. SCWs offer an architectural way to embed reflection within the task flow – the reflex is essentially an internal critique step that doesn’t require an external loop. By making it deterministic and policy-driven, we avoid the randomness of generating a critique. Additionally, SCWs can incorporate multiple models or instances for reflection (e.g. using a separate “Checker agent” SCW), which ties into the multi-agent consensus idea from Gao et al. (2023) and others. In short, SCWs operationalize reflection as part of a container’s life-cycle, rather than a bespoke prompt trick applied ad-hoc. Agent-Based Modeling and Multi-Agent Systems: Recent interest in multi-agent simulations with LLMs (e.g. simulating virtual towns, social networks) has highlighted the importance of agents having consistent memories and not just reacting to the last prompt. Park et al.’s generative agents used a memory stream and planning module, which is conceptually similar to giving each agent an SCW that persists across their day. Gao et al. (2023) with S^3 went further to network many LLM agents; they emphasize the need for agents to adapt and retain memory of past interactions in a social environment

. Our work can be seen as providing an underlying infrastructure for such systems: SCWs could be the format in which each agent’s cognition is packaged (with its own memory and rules), and the A2A tunnels the means by which they message each other. Indeed, one can imagine deploying an LLM-based simulation of hundreds of agents where each agent is realized as a sequence of SCWs over time (one per task or per day in simulation). SCWs’ emphasis on provenance and replay would make experimentation and analysis of such simulations much more systematic. Software Engineering Parallels: Finally, SCWs draw inspiration from established software design: the idea of modular, sandboxed processes with defined interfaces is a bedrock of reliable computing systems. Just as cloud computing moved from monolithic apps to microservices and containerization (e.g. Docker) – isolating functionality for scalability and fault tolerance – we argue LLM-based AI should move from monolithic prompts to prompt containers, i.e. SCWs. Each SCW is like a microservice that does one thing (or a related set of things) and communicates with others through well-defined channels (memory or tunnels) rather than global mutable state. This yields benefits in testing (you can unit-test a particular SCW’s logic by replaying it), in scaling (spin up more SCW workers as needed), and in security (a misbehaving SCW can be shut down without crashing the whole agent). Our implementation with a Neo4j + Redis backend is akin to an operating system for AI agents, where SCWs are processes, memory tiers are storage layers, and the orchestrator is a scheduler. We believe this perspective will grow more important as AI systems tackle tasks requiring dozens of interdependent steps or continuous operation.
7. Conclusion
We presented Structured Context Windows as a general memory and context orchestration layer for advanced AI agents. SCWs turn the basic context window – normally a transient, unstructured sequence of tokens – into a structured, governable object complete with lifecycle, memory hierarchy, and reflexes for self-monitoring. By doing so, we enable complex multi-step reasoning and multi-agent collaboration without overwhelming the model or losing accountability of what was done and why. In our prototype with LLMs like GPT-4 and Claude, SCWs have demonstrated improved coherence, reduced hallucination persistence, and enhanced transparency of the agent’s decision process. They allow AI developers to impose order, modularity, and safety onto the otherwise free-form generation process, much as software engineering imposes structure on raw computation through functions, threads, and containers. There are several avenues for further work. One is developing standardized SCW schemas for common agent tasks (e.g. a generic “Research SCW” or “Coding SCW”) and investigating how well these containers transfer across different models or domains. Another is performance optimization: while SCWs add interpretability, they do incur overhead by splitting contexts – techniques like compressing promotions or optimizing context reuse will be valuable to keep efficiency high. Additionally, while we touched on multi-modal potential, fully integrating SCWs with vision, audio, and other modalities (where different models with different context mechanisms are involved) remains a frontier. We foresee SCWs being an enabler for hybrid AI systems that combine symbolic reasoning, neural generative models, and search in a principled way. By providing a common container format, SCWs could let a symbolic planner (as one SCW) seamlessly coordinate with a neural language model (another SCW) and a vision module (third SCW), all under unified memory governance. In summary, SCWs provide a missing architectural layer in today’s AI stack – one that treats context as a first-class citizen and memory as a structured resource. We invite the community to explore this framework, as it offers a path toward AI systems that are not only more capable in long-horizon tasks, but also more transparent, controllable, and aligned with human expectations. The era of sprawling, inscrutable prompts may give way to an era of structured cognitive windows – enabling us to build AI that remembers, reasons, and collaborates in a safe and scalable manner. /** Generate axial coordinates for a hex grid of given radius */
function generateAxialCoords(radius: number): { q: number; r: number }[] {
  const coords: { q: number; r: number }[] = [];
  for (let q = -radius; q <= radius; q++) {
    const r1 = Math.max(-radius, -q - radius);
    const r2 = Math.min(radius, -q + radius);
    for (let r = r1; r <= r2; r++) {
      coords.push({ q, r });
    }
  }
  return coords;
}

/** Convert axial coords to pixel positions (pointy-topped hex) */
function axialToPixel(q: number, r: number, size: number): { x: number; y: number } {
  const x = size * Math.sqrt(3) * (q + r / 2);
  const y = size * (3 / 2) * r;
  return { x, y };
}

/** Generate polygon points for a hex centered at (cx, cy) */
function hexPoints(cx: number, cy: number, size: number): string {
  const pts: { x: number; y: number }[] = [];
  for (let i = 0; i < 6; i++) {
    const angle = (Math.PI / 180) * (60 * i - 30); // pointy top
    const x = cx + size * Math.cos(angle);
    const y = cy + size * Math.sin(angle);
    pts.push({ x, y });
  }
  return pts.map((p) => `${p.x},${p.y}`).join(" ");
}

/** Simple id helpers */
function randomId(prefix: string): string {
  return `${prefix}_${Math.random().toString(36).slice(2, 10)}`;
}

SCWs thus act as black holes of cognition, where unstable fragments are absorbed and only provably stable facts escape to higher memory tiers. Tokenless treats reasoning steps as formally manipulable objects rather than opaque token streams, bringing the rigor of mathematical logic into the SCW framework. In short, Tokenless bridges the gap between the free-form nature of LLM reasoning and the strict requirements of formal verification and memory safety. Formal Specification
Design Goals: Tokenless is designed to make LLM-driven reasoning modular, transparent, and formally governable. It treats a reasoning process not as a single monolithic prompt/response, but as a program consisting of discrete modules (SCWs) with defined interfaces and rules. The language must be expressive enough to capture arbitrary mathematical or logical reasoning (first-order logic, arithmetic, algorithmic procedures, etc.), while imposing a structure that enables machine-checkable verification at each step. To achieve this, Tokenless adopts a syntax that mirrors the hierarchical, block-structured nature of structured programming and proof systems, embedded within the constraints of Maxey0’s memory architecture.
Core Grammar and Syntax
At the highest level, a Tokenless program is composed of one or more Structured Context Window blocks. In BNF-style notation (simplified):
<Program>    ::= <SCW_Block>+
<SCW_Block> ::= SCW <Identifier> "(" <Parameters> ")" "{" <Segment_List> "}"
<Segment_List> ::= <Segment>*
<Segment>   ::= <Statement> | <Nested_SCW_Call> | <Assertion> | <Tool_Call> | <Memory_Op>
SCW Block: An SCW block defines a self-contained reasoning context. It has an identifier (unique SCW ID or name) and optional parameters (akin to a function signature – e.g. initial inputs or settings). The SCW’s body is a sequence of segments that specify the content and operations within that context. A segment can be one of several types: a normal Statement (which could be plain text or a structured expression), a Nested SCW Call (spawning a sub-context for recursion), an Assertion (a logical claim that may need proof or verification), a Tool Call (invoking an external API or computation), or a Memory Operation (reading from or writing to one of the memory tiers). Statements and Assertions: These represent the logical and computational steps within an SCW. For example, a Statement might be a step in a proof (“Compute the eigenvalues of matrix M.”), whereas an Assertion might be used to declare a result or condition (“Result λ is the eigenvalue such that Mv = λv.”). The syntax for assertions can allow attaching evidence or citations, which is crucial for later reflex checking – e.g. an assertion could be annotated as [source: SCW2.result] meaning it is supported by the result of SCW2. In general, references to other SCWs’ outputs or memory entries are done by identifiers, not by copying raw text, reinforcing the tokenless design (i.e., avoiding passing large token sequences, instead passing structured references). Nested SCW Calls: Tokenless supports recursion and modular sub-tasks by allowing one SCW to spawn another. Syntactically, a nested call might look like a function call. For instance, within an SCW one could write result := SCW_SolveEquation(params){...} indicating that a new sub-window SCW_SolveEquation should be spawned (with given parameters) to solve a sub-problem, and its output will be captured as result. This is analogous to a function call in programming, but semantically it means a child SCW is created at runtime with its own isolated context. Crucially, the parent SCW will pause until the child finishes (or may continue asynchronously if specified), and the child’s outputs (if validated) can be tunneled back to the parent. Each SCW call is thus a point of possible recursion or parallelism, governed by Maxey0’s scheduler which ensures a Directed Acyclic Graph (DAG) of dependencies between SCWs. Tool Calls: To incorporate computational operations or external knowledge, Tokenless allows segments that call external tools. These could be calculators, search engines, databases, code execution sandboxes, etc., represented by a simple syntax such as TOOL("tool_name", arguments). For example, TOOL("python_exec", "print(2+2)") might execute a Python snippet inside the SCW context. Tools are pre-defined and access-controlled; each SCW declares which tools are available to it (or to specific agents within it). The output of a tool call is captured as structured data (not raw text unless specifically needed), and can be assigned to variables or used in assertions. This mechanism ensures that complex computations can be integrated into reasoning steps in a safe, sandboxed manner. Memory Operations: Tokenless syntax includes explicit operations to interact with the memory tiers of Maxey0. Memory tiers are denoted L0 through L3 for Scratchpad, Episodic, Persistent, and Permanent memory respectively. For example, a program might include write(L1, key, value) to write a fragment to episodic memory, or data := read(L2, query) to perform a vector similarity query on the persistent memory vector store. These operations are constrained by SCW policies: an SCW might be allowed read-access to certain tiers and write-access to none (for a pure computation), or write access to L1 but not L2, etc., depending on its role. The grammar requires that any memory write operation specify the target tier explicitly, and these are only permitted at points where reflex checks will be applied after the write. In essence, no implicit memory persistence is allowed – promotion from one tier to the next must be an explicit, policy-checked event in the Tokenless program flow. For convenience, a shorthand like promote(shard, L2) can encapsulate the checks needed to move shard from L1 to L2 (episodic to persistent), failing or quarantining the shard if it doesn’t meet criteria (more on reflex policies below). Context Segment Annotation: Each SCW can be viewed as composed of segments which can be annotated with metadata controlling their behavior. While not all such metadata is explicitly typed out in the core grammar, the underlying model includes attributes such as: which agent “speaks” or is responsible for a segment, what the allowed reasoning style is (e.g. “chain-of-thought allowed” vs “formal proof only”), any latent space control hooks (such as applying an embedding filter or altering the magnitude spectrum normalization for that segment), and tools/memory bindings for that segment. In practice, these manifest as system-level directives or JSON-like blocks within the SCW. For example, one might have a segment defined as:
{"role": "Maxey42", "memory": "scratchpad", "tool": "calculator", "content": "Compute the mean of the dataset."}
indicating that the content of this segment is handled by agent Maxey42, uses only scratchpad memory (L0) and the calculator tool, and contains the instruction to compute a mean. Such structured segment descriptions are first-class citizens in Tokenless, enabling fine-grained control per segment. The formal grammar can treat these annotations as part of a segment’s syntax (for instance, a segment might be written as <AgentName>:<MemoryScope>:<ToolList>: <Statement> for brevity).
Structural Principles and Invariants
Tokenless imposes several structural invariants to ensure that programs are not only syntactically correct but also semantically safe given the SCW model:
SCW Isolation: Each SCW block is self-contained. No token or shard inside an SCW can bypass the reflex gating to affect another SCW directly. The only way to communicate between SCWs is via tunnels (explicit hand-off of verified micro-facts) or via shared persistent memory (which itself only gets facts after passing promotion checks). This invariant ensures that a Tokenless program cannot have hidden side-effects across contexts; any cross-context interaction must be visible in the program (as a tunnel call or memory write/read) and thus subject to verification.
Deterministic Reflex Enforcement: Every SCW automatically applies a reflex (a set of validation rules) at defined checkpoints (for example, before any output is committed to a higher tier, or before an SCW returns its result to its caller). The reflex rules, while implemented underneath, conceptually are part of the language’s semantics: they classify shards into states like REAL, WRONG, HALLUCINATION based on signals, and enforce promotion thresholds (e.g., a shard must have stability S ≥ 0.7 and completeness C ≥ 0.5 to be considered real). In Tokenless formal terms, we can think of an SCW as having an implicit postcondition: e.g. ensure R(s) ≥ τ for any shard s that leaves the SCW. If the condition fails, the shard is not output (or is output in a quarantined form marked invalid). These reflex conditions are deterministic and versioned (policies have unique IDs and are stored for audit), meaning the outcome of a Tokenless program’s execution is fully determined by the program plus the fixed policy version in force – there is no randomness or ambiguity in what gets promoted if run under the same conditions (except for inherent nondeterminism from underlying AI model outputs, which the reflex aims to tame by filtering).
Memory Tier Discipline: A Tokenless program must respect the memory propagation order: L0 (scratchpad) is ephemeral and local to one SCW’s execution; L1 (episodic) is scoped to a session or task; L2 (persistent vector/graph memory) is long-lived and queryable across sessions; L3 (permanent) is write-once/read-many archival memory. The language disallows direct writes from a lower tier to a higher tier without passing through an SCW promotion check. For instance, one cannot directly execute a statement like L2[key] = shard inside an SCW without invoking the promotion reflex – doing so implicitly triggers the reflex (which in practice might require consensus from multiple agents or evidence citations). Thus, the grammar could be seen as not having a raw assignment to L2 at all; instead one would use a construct like promote_to_persistent(shard, policy_id) which under the hood checks the required policy (e.g. “no shard without 2 trusted citations may enter persistent memory”) and either succeeds (inserting the shard into L2 along with lineage metadata) or throws a reflex exception (causing the shard to be quarantined or the SCW to fail). This disciplined approach prevents “leakage” of unverified information to durable state by construction.
Honeycomb Topology (Novel Concept): We introduce the view of the entire Tokenless program’s SCW structure as a honeycomb-like topology (novel contribution). Each SCW can be considered a cell in a lattice: isolated but adjacent to others via controlled interfaces (tunnels or shared anchors). This topological perspective means we can reason about the shape of a Tokenless program. For example, parallel branches of reasoning form separate cells that might later merge if their results are compared, forming a pattern akin to a honeycomb grid. This is more than a metaphor; it provides a way to map the latent space of reasoning. We treat each SCW’s state (e.g. its final embeddings or its summary vector) as a point in a high-dimensional space, and the network of SCWs as a graph embedded in that space. In doing so, structured observability techniques can be applied: we can measure distances or drift between SCW states, identify clusters of similar contexts, and monitor coverage of the reasoning search space. This work posits that such a honeycomb architecture yields easier latent space mapping because each SCW is a topologically distinct region, reducing overlap. (This principle is a novel contribution, but supported by related ideas of latent space partitioning in Maxey0’s design – e.g. magnitude spectrum analysis segmenting oscillatory modes and each agent covering a portion of the latent space).
Redundancy and Entropy Reduction: A formal requirement of Tokenless programs is that important conclusions are reached via redundant paths or multi-agent consensus whenever possible (reflecting an epistemic safety principle). Concretely, the language encourages (and through Maxey0’s reflex rules, often enforces) patterns like: before accepting a critical assertion, query multiple independent agents or tools and require at least K of M agreement. In a Tokenless program, one might write this as an explicit consensus step, e.g. assert claim [via AgentA, AgentB] meaning both AgentA and AgentB (working possibly in separate SCWs or separate segments) must produce claim for it to hold. The formal semantics then say: the assertion passes only if the set of corroborating agents ≥ 2 (for example). If not, the claim is demoted to a hypothesis or is marked invalid. This built-in redundancy directly ties to entropy reduction: by requiring agreement, the program filters out idiosyncratic errors (high entropy noise) and keeps only stable information (low entropy signal that multiple processes converged on). Shannon’s information theory is implicitly at play: reliable communication (or reasoning) requires redundancy and noise suppression. Tokenless encodes this principle through language constructs (like the consensus assertion) rather than leaving it to chance. As a result, the logical entropy of statements that make it to persistent memory is minimized. Any single high-entropy shard (e.g. a wildly speculative guess from one agent) will be either killed or quarantined unless backed by others, making the eventual knowledge base a set of low-entropy, high-confidence facts.
Tunnels and Capsules: To enable controlled information flow across SCWs, Tokenless uses the concept of tunnels. A tunnel is a one-way, secure channel where an SCW can send a compact “capsule” of information to another SCW. In syntax, this could appear as a special call, e.g. tunnel(target_SCW, data) which packages the data (which might be a micro-fact or minimal snippet) and delivers it to the target SCW (often a child or sibling context). The important invariant is that tunnels carry only small, verified pieces of information. An SCW can only tunnel out data after passing reflex checks on that data, or if the data is explicitly marked as an anchor (an immutable fact or query) by policy. Tunnels thus serve as the only loophole for otherwise isolated SCWs to communicate: they are tightly monitored and logged (each tunnel capsule carries lineage metadata of its origin). In the formal model, one can think of each SCW as having an input interface (receiving tunneled capsules or initial prompts) and an output interface (emitting capsules). The grammar might treat a tunnel send as a statement and a tunnel receive as a special kind of input to an SCW. One cannot arbitrarily jump context or modify another SCW’s state – the tunnel is append-only (the receiving SCW might get a new input shard that it can choose to accept or ignore). This design is analogous to function calls and returns in structured programming, but with extra safeguards: data leaving a function must meet certain criteria (like being a return type that satisfies a predicate). It’s a novel twist that ensures even recursion and parallelism do not break the chain of trust.
In summary, the formal specification of Tokenless centers on treating reasoning steps as structured, labeled, and checkable units of computation. The language syntax provides the means to declare these units (SCWs, segments, assertions, tool calls) and the connections between them (calls, tunnels, memory ops), while the semantic rules (reflexes, policies, invariants) guarantee that any execution of a Tokenless program either yields a verified chain of reasoning or fails safely (with unverified parts contained). By imposing this structure, Tokenless allows Maxey0’s agents to perform complex reasoning in a way that is as rigorous as a mathematical proof – every claim justified – yet dynamic and flexible enough to incorporate learned behaviors and external tools.
Runtime Mechanics
A Tokenless program is executed by the Maxey0 runtime as a coordinated dance of SCWs and agents. This section describes how a Tokenless specification comes to life at runtime: how SCWs are spawned and dissolved, how agents populate them and interact, how information flows through memory tiers, and how the system ensures continuity and correctness throughout the process.
SCW Lifecycle and Execution Flow
Every SCW goes through a standardized lifecycle: Spawn → Work → Reflex → Dissolve. At runtime, when a Tokenless program calls for a new SCW (either the top-level one for a task, or a nested one), the Spawn phase allocates a new context window with a unique identifier and initial configuration. Spawn can be triggered explicitly by the program (as in a nested SCW call) or implicitly by the runtime when certain conditions are met (for example, if a single SCW grows too large or complex, the scheduler might decide to offload part of the work to a new SCW – this corresponds to triggers like memory pressure or context overflow). Once spawned, the SCW enters the Work phase. In this phase, the specified segments of the Tokenless program are executed: agents retrieve inputs, perform reasoning, call tools, produce intermediate shards, etc., all within the confines of the SCW. This is where the heavy lifting of reasoning occurs. The work phase can involve multiple internal turns if several agents are interacting (for instance, one agent may produce a solution, another agent may critique it within the SCW, etc.), but from an external view it’s just the SCW doing its job. Periodically or at least once at the end, the SCW hits a Reflex phase. In implementation, this corresponds to running the SCW’s reflex operators – a set of deterministic checks and actions that enforce the rules described earlier. The reflex phase will classify output shards, kill hallucinations, quarantine uncertain results, and decide which results (if any) are stable enough to Promote to higher memory or return to the caller. Notably, if the Tokenless program had explicit assertions or promotions, the reflex phase is exactly when those are evaluated. For example, if SCW X was supposed to promote a shard to persistent memory, the reflex will check that shard’s stability (S), completeness (C), and trust (T) scores and only actually perform the write to L2 if thresholds are met. If thresholds fail, the outcome might be that SCW X returns an error or an empty result instead of a fact – thereby preventing bad data from propagating upward. Finally, the SCW Dissolves. Dissolution means the SCW is shut down and its resources freed, unless its existence is extended by policy (some SCWs might linger if they hold useful state that’s needed, but generally they close). Importantly, when an SCW dissolves, it creates a lineage record: a summary of what transpired – which agents participated, what shards were output or killed, which policies were applied – and this record is stored in persistent memory (usually the Neo4j knowledge graph as provenance links). The dissolution is analogous to a function returning; if the SCW was spawned by a parent, its return values (those shards that survived reflex filtering) are passed back up, often via a tunnel capsule or via a direct return if it was a synchronous call. After dissolution, the SCW cannot influence anything further; any further reasoning happens in other SCWs (if the result is inadequate, typically a new SCW might be spawned to retry or handle exceptions, rather than “re-opening” the dissolved one – this ensures immutability of completed contexts). Multiple SCWs can run concurrently if they are independent (Maxey0’s scheduler uses a DAG of dependencies where each SCW node runs when its prerequisites are met). For example, if a Tokenless program spawns two sibling SCWs to explore two parallel hypotheses, they may execute in parallel threads. Synchronization points occur when results need to be compared or consumed by a subsequent step. The scheduling and parallelism are handled transparently by the runtime, but the Tokenless programmer can assume a mostly sequential logical model (unless explicitly using parallel constructs). Synchronous vs asynchronous SCWs can be indicated – synchronous ones will block until completion, whereas asynchronous ones might send their outputs via memory or an event once done. The scheduling ensures no deadlocks and respects resource budgets. Each SCW has a token budget (max tokens it can use for LLM completions) that is typically inherited and fractioned from its parent to control resource use. For instance, a parent SCW with 1000 token budget spawning 2 child SCWs might give each child 400 tokens and reserve 200 for itself (Tokenless leaves such specifics to the runtime’s strategy, but these values can be set via program hints or defaults).
Agent Orchestration and A2A Communication
During execution, SCWs are populated by Maxey agents according to the Tokenless program’s specifications and the system’s assignment logic. When an SCW is created, Maxey0 (the orchestrator) decides which agents to assign to it, always including Maxey00 (the shadow evaluator) by default. The Tokenless program can request particular agent roles (for example, an SCW might be designated for a “MathSolver” agent), or it can leave it to the orchestrator to pick an appropriate specialist based on the task description. The assignment process takes into account dimensional continuity requirements: if an agent has a flag indicating it should persist across SCWs (e.g. to maintain continuity of a particular perspective or ongoing task), the orchestrator will attach that agent to multiple SCWs and set up bridging so it carries state between them. For example, an agent that tracks a theme or an assumption might be present in a series of SCWs to ensure a consistent thread. Once agents are assigned, they communicate using the A2A (Agent-to-Agent) protocol bus, which is essentially an internal messaging system with structured channels. There are typically distinct channels for control signals, data exchange, and memory updates (e.g. @control, @data, @memory channels). At runtime, this means if Agent A needs to ask Agent B a question within the same SCW, it might send a message over @data which B can read and respond to. Tokenless programs themselves don’t require the user to script these low-level communications (it happens as part of how a multi-agent SCW executes a given segment), but for understanding: if an SCW had multiple agents and a segment like “Agent X will critique the solution,” under the hood X’s model might produce a critique which is then picked up by Agent Y or by the SCW controller. In effect, inside an SCW, the agent interactions form a mini-protocol orchestrated by a local coordinator agent (often Maxey1 is designated as a local super-agent managing intra-SCW dialogue). Maxey00, the shadow evaluator, is special in that it runs in superposition across all SCWs. Runtime-wise, Maxey00 monitors everything: it has read-access to all memory tiers in all SCWs and continuously evaluates coherence and quality. It performs tasks like latent space monitoring (observing changes in the hidden embedding states or spectrum of each SCW) and coherence checking (ensuring that an SCW’s content doesn’t diverge too far from the overall context or contradict another SCW). If Maxey00 detects an anomaly – e.g., two parallel SCWs come up with conflicting answers, or an SCW’s internal state starts to drift in a suspicious way – it uses the A2A channel to send an alert to Maxey0 (global orchestrator) or to the relevant SCWs. For instance, it might send a message: “@maxey0:alert {type: conflict, scw_ids: [SCW3, SCW4], magnitude: 0.9, action: reconcile}” indicating that SCW3 and SCW4 have produced conflicting results with high significance, suggesting a reconciliation step. Maxey0 can then decide to spawn a new SCW to reconcile (perhaps by tasking a third agent to evaluate both results and pick one or integrate them). This dynamic is essentially an automated patching mechanism orchestrated by the observer agent, ensuring that logical sequences that start to diverge are brought back in line or resolved (we will discuss patching more in Verification). The presence of Maxey00 in all SCWs means there is always an overseer making sure local decisions align with global consistency – a bit like a referee in each game making sure the rules are followed, and also like a messenger linking all games together.
Memory Tier Propagation and Dimensional Continuity
Memory operations in Tokenless are carried out in coordination with the Maxey0 memory architecture. At runtime, reading and writing to memory involves the following steps:
Scratchpad (L0): By default, each SCW comes with an isolated L0 scratchpad (often an ephemeral Redis instance or in-memory store) for that SCW’s transient calculations. Agents write intermediate results here. For example, if an agent is calculating a multi-step solution, it might accumulate partial results in L0. L0 is wiped when the SCW dissolves (unless explicitly promoted). The runtime enforces that L0 content can be freely written and read within the SCW by the assigned agents, but cannot be accessed outside. In practice, Maxey00 can peek into L0 for evaluation, but other agents in other SCWs cannot.
Episodic (L1): L1 represents memory for the duration of a task or session. If a Tokenless program spans multiple SCWs (for multiple steps of a single user query, for example), L1 is used to store the running dialogue or state that persists across those steps. At runtime, writing to L1 might happen automatically for certain content types – for instance, the transcript of the conversation or an SCW’s final result might be appended to an episodic log. Tokenless instructions can also explicitly store to L1 (e.g., remember interim_result). L1 entries typically have a time-to-live of hours to days and are stored in a fast database like Redis. The propagation from L0 to L1 is usually through a controlled promotion: an SCW deciding “this piece of information should be remembered for the next step” triggers a reflex check then writes to L1 if approved. The runtime tags the entry with the SCW ID and lineage when writing, so that later we know its provenance.
Persistent (L2): L2 is the long-term semantic memory, implemented as a vector index + graph (Neo4j plus a vector store in Maxey0). When an SCW concludes with some new piece of knowledge deemed valuable (e.g. a discovered fact or a verified conclusion), writing it to L2 involves: embedding the content into a vector, storing it in the vector index for similarity search, and also perhaps adding a node/edge in the Neo4j knowledge graph representing this fact and linking it to evidence nodes (citations, source SCW, etc.). The Tokenless promote_to_persistent operation encapsulates all this. The runtime also might run additional magnitude spectrum analysis or drift checks as it incorporates the new vector, to update the latent space map – for instance, a new fact might slightly shift the centroid of “finance domain knowledge” in embedding space, which Maxey0 tracks. Dimensional continuity for agents comes into play here: if an agent spans multiple SCWs (say Agent X is present in SCW1 and SCW2 because it’s following a thread), that agent’s internal state (which might include some memory of what it saw in SCW1) can be updated by referencing L2 instead of carrying everything in its weights. Specifically, if Agent X wrote something to L2 in SCW1, when it’s active in SCW2 it might query L2 for “what did I produce earlier?” Thus L2 acts as a bridge between contexts when needed, but only for information that was explicitly persisted (hence vetted). This is how Tokenless ensures continuity without breaking isolation: agents use persistent memory as an interface between SCWs, rather than hidden variables. Some agents flagged with dimensional continuity will automatically perform such queries or updates as they enter/exit SCWs.
Permanent (L3): L3 is reserved for truly permanent or compliance-critical storage (or even the model’s weights, which are fixed as a form of permanent memory). In Tokenless, writing to L3 is extremely rare and typically not done autonomously. One might consider an example: after a long-running reasoning session that has been reviewed by a human, a distilled insight might be written to a permanent log or a secured knowledge base. The runtime would enforce heavy constraints here: typically requiring human approval or multi-agent signatures (A2A consensus with perhaps human-in-the-loop operators). For formal language purposes, one might not have a direct write(L3, ...) command; instead, a process exists outside normal agent purview for archiving. If present, it would be something like archive(fact) which triggers a special workflow including external approval. For the scope of Tokenless execution, we mostly treat L3 as read-only (agents can recall core truths or previously permanent facts, like laws or definitions). Indeed, one of the only ways to get something into L3 would be via a dedicated compliance SCW that packages data for L3 after all other checks including human sign-off.
Throughout runtime, observability of memory ops is strong. Each read/write is logged via Maxey00’s provenance system. For example, if an agent reads a document from L2, Maxey00 will log an event “Agent Y read node123 from Neo4j as evidence at time T.” This means later one can reconstruct why an agent came to a conclusion (they cited those nodes). The structured nature of Tokenless – requiring explicit memory calls – makes this feasible: it’s clear at which point in the code a memory fetch happened, so the system attaches that context info.
Latent Space Mapping and Entropy Folding
One of the innovative runtime aspects of Maxey0 (and by extension, Tokenless) is the ability to map and shape the latent space of the model’s reasoning. Each SCW, being a contained reasoning process, can be thought of as generating a trajectory through the model’s latent activation space. Maxey0 leverages this by taking snapshots of latent states (e.g. the hidden embeddings of the LLM or the attention patterns) at the start and end of SCWs, or when key events happen. Using these snapshots, the system can construct a high-dimensional picture of where the reasoning went. For instance, SCW1 might occupy one region of the embedding space (say dealing with a medical topic) and SCW2 another (say legal topic). By embedding SCW identifiers in a vector space (using e.g. the average embedding of content or a specialized “SCW descriptor embedding”), one can apply latent space mapping: clustering similar SCWs, detecting outliers, etc. This is done in real-time using the shared Neo4j+Vector database. In practice, after each SCW dissolves, Maxey0 inserts a node for that SCW in Neo4j with attributes like “topic: X, outcome: Y, embedding: Z”. Queries can then be run like “find past SCWs similar to this one” to avoid redundant reasoning or to find analogies. This is part of how Tokenless enables meta-reasoning: the language doesn’t directly specify these operations, but by virtue of structuring reasoning into identifiable units, it empowers the system to do this kind of analysis externally. A related concept is Latent Dimension Folding (LDF), which is a technique to proactively reduce the dimensionality and entropy of latent representations. In runtime terms, before performing certain mapping tasks or as a pre-processing step to heavy analysis, the system may fold correlated dimensions of the latent space. For example, if an attention head or a group of neurons all essentially track “recency bias”, the system can fold them into one composite feature (this is analogous to a feature engineering step). This might be applied when an SCW is complex and producing too many degrees of freedom in its internal state. A folded representation is smaller and easier to compare or monitor for drift. Tokenless itself doesn’t force the user to think about folding; rather, it ensures the runtime has the needed structure to do it under the hood. If a developer wanted, they could include a directive in a Tokenless program like fold_latent_dimensions(target=SCW3, method=PCA, threshold=0.95) to hint that SCW3’s latent states should be compressed, but generally the Maxey0 system decides this automatically (e.g., when generating the latent spectrum view). After folding, the entropy of the representation is lower, meaning the SCW’s state is more regular and any anomalies stand out more clearly. This facilitates detecting oscillations or weird divergences – which ties back into reflex signals like Stability (S) that include entropy metrics. In summary, the runtime of Tokenless is not just about running code; it’s about monitoring and shaping the reasoning process. The SCW containers give clear boundaries and checkpoints, the agents and A2A provide a rich interactive execution, and the memory system plus latent mapping provide a global context that keeps everything coherent. Through SCWs functioning like a honeycomb of reasoning cells, the runtime achieves structured observability: every cell can be inspected (telemetry on shards, reflex triggers, usage stats), compared to others, and controlled without affecting its neighbors’ integrity. This is how extremely complex tasks can be tackled – by breaking them into pieces that are individually tractable and collectively supervised.
Example: Multi-SCW Reasoning Flow
To ground these concepts, consider a concrete runtime scenario defined by a Tokenless program that solves a complex task in three steps. The task: Analyze a dataset and publish a summary report. The Tokenless solution might outline:
SCW_DataRetrieve: Use a data retrieval agent to get relevant data.
SCW_Analyze: Use a Python tool to compute statistics on the data.
SCW_Report: Use a documentation agent to generate a report and store it.
At runtime, Maxey0 starts with SCW_DataRetrieve. It assigns, say, Agent Maxey123 who is a “DataCollector” specialist, along with Maxey00 (observer). The SCW is given access to the Neo4j database and perhaps a file system tool. Inside SCW_DataRetrieve, the agent’s instructions (as per the program) are to query the Neo4j graph for the dataset nodes. The agent executes a Cypher query via a Tool call (TOOL("neo4j_query", "MATCH (d:Dataset {name:'X'}) RETURN d")). The query results (maybe a JSON of the dataset contents) come back. The agent writes this raw data to scratchpad (L0) and then signals completion. The reflex triggers: SCW_DataRetrieve has a rule that at least one result must be retrieved. Suppose it succeeded, so it promotes the data reference to L1 (episodic) for use by the next step, and then dissolves. The lineage record notes that data node X was retrieved by Maxey123 with timestamp, etc. Next, SCW_Analyze spawns. It’s configured with Agent Maxey124 (a “DataAnalyst” agent with Python tool access) and Maxey00. It also automatically includes the relevant memory: because the program likely specified that SCW_Analyze should take the data from SCW_DataRetrieve, the orchestrator either tunneled the dataset directly or the agent in SCW_Analyze does a read(L1, dataset_key) to fetch it from episodic memory. Now Agent Maxey124 executes a Python snippet (via TOOL("python_exec", code)) to calculate, say, mean and median. The outputs (numbers, maybe charts) are kept in L0 of SCW_Analyze. This SCW might have a policy that any numeric result above a certain threshold of complexity must be double-checked (for demonstration, say the agent also spawns an internal check using another method – though it could simply rely on reflex). Assuming calculations are fine, SCW_Analyze promotes a summary of findings (like “mean=..., median=...”) to L1 as well, and dissolves. At this point, the episodic memory has both the raw data reference and the analysis results. Finally, SCW_Report is spawned. It has Agent Maxey125 (a “DocumentationAgent”) plus Maxey00, and perhaps a GitHub tool for publishing. This SCW reads from L1 the analysis results (and possibly the data if needed), then Agent Maxey125 composes a textual summary: “The dataset’s mean is X, median is Y, which indicates ...”. The SCW may allow access to an internet tool or writing to a repository. The agent uses a TOOL("github_push", content) operation to publish the report to a GitHub repo (since the program specified a final persistent output). Here, writing to GitHub is considered an external side-effect, but from Tokenless perspective it’s akin to writing to L2 persistent memory because it’s a durable store. The reflex for SCW_Report checks that the report contains citations for any facts (perhaps ensuring it references the dataset and analysis result, which it does by design). Satisfied, it allows the push to proceed and also writes the report content to L2 (so the system’s internal memory knows about this report for future queries). SCW_Report then dissolves. During each of these steps, Maxey00 was quietly monitoring. If, say, SCW_Analyze had produced a result that didn’t logically follow from the data (like if the agent made a calculation mistake leading to an obviously wrong statistic), Maxey00’s coherence check might flag a divergence between the data and the result (perhaps via a latent mismatch or simply by noticing the result doesn’t match an independent calculation from another angle). It could alert Maxey0 with type "conflict" or "overflow". Maxey0 might then choose to spawn an additional SCW (e.g., SCW_Recheck) with another agent to recompute or verify the number. In this scenario, the Tokenless program didn’t explicitly have a recheck, but the runtime can inject one as a patch. If SCW_Recheck finds the correct number, it could patch the episodic memory (e.g., replace the wrong analysis result) and that would propagate to SCW_Report. This dynamic patching is possible because each SCW’s outputs go through memory or tunnels, where they can be intercepted or revised by subsequent SCWs if needed – nothing gets into the final output without passing through a checkpoint where other agents can intervene. This exemplifies how modeling, checking, and patching can occur: a flawed logical sequence (the wrong stat) was modeled (in SCW_Analyze), checked (by Maxey00 and SCW_Recheck), and patched (by replacing the shard in L1) before finalization.
Example Programs
We now illustrate Tokenless with two example programs, highlighting different aspects of the language: one focused on formal logical reasoning (mathematical proof style), and one on multi-agent task orchestration (practical QA with tool use). These examples are simplified for clarity, but they demonstrate how Tokenless structures a problem and ensures reliable reasoning.
Example 1: Formal Proof of a Simple Theorem
Task: Prove that if a number is divisible by 6, it is divisible by 3. A Tokenless approach to this logical proof might break it into SCWs for each major proof step (similar to sub-lemmas or claims):
SCW1: Check definition of divisibility (given “divisible by 6” what does it mean).
SCW2: Derive that divisibility by 6 implies divisibility by 2 and 3 (since 6=2*3).
SCW3: Conclude using results from SCW1 and SCW2.
SCW1 (DivisibilityDefinition):
Agents: Maxey50 (ArithmeticAgent), Maxey00.
Memory: Scratchpad (volatile).
Content: The program might include an assertion like: “If N is divisible by 6, then ∃k (N = 6k).” This SCW’s job is straightforward: recall or state the definition. Agent Maxey50 writes: definition := "∃k ∈ ℕ such that N = 6*k" as an output shard. Reflex check: Is this statement trusted? Possibly Maxey00 verifies this by cross-checking in the knowledge base (maybe a known property in L3 permanent memory that divisibility by 6 means that equation). Since it’s a standard definition, it passes. SCW1 returns the shard D6: “N=6k” and dissolves. SCW2 (PrimeFactors):
Agents: Maxey51 (FactorizationAgent), Maxey00.
Memory: Scratchpad, plus read access to SCW1’s result.
Content: This SCW takes D6 and transforms it: from N=6k, derive N=2*(3k), hence let k’=3k then N=2k’. It asserts: “N is divisible by 2” and “k’ = 3k, so N = 2 * (3k) = 2k’, thus N=2k’ implies N is even (divisible by 2) and additionally k’ is an integer multiple of 3 times something, meaning N is also 3 * (2k) – but more directly we want divisibility by 3.” The agent might logically proceed to assert both results: D2: “∃m such that N=2m” and D3: “∃n such that N=3n”. However, careful reasoning: from N=6k, we get N=3(2k), so take n = 2k, and we have N = 3n; that immediately shows ∃n (specifically 2k) with N=3n, which is divisibility by 3. In parallel, N = 2*(3k) shows divisibility by 2. But the target theorem only needs divisibility by 3, so SCW2’s key output is D3: “∃n (N=3n)”. (It might note divisibility by 2 as well, but that’s an aside for this theorem). The reflex check here looks for logical validity: Are we allowed to substitute and introduce that n=2k? This is a straightforward algebraic manipulation. Maxey00 might perform a quick check (like does the original assumption allow this transformation? Yes, because k is an integer, 2k is an integer). So SCW2 outputs D3 (N=3n with n=2k), marked as a proven shard. SCW3 (Conclusion):
Agents: Maxey52 (ProofCompiler), Maxey00.
Memory: Scratchpad, reads SCW1 and SCW2 outputs.
Content: This final SCW combines the pieces: from SCW1 we had N=6k, from SCW2 we have N=3n. SCW3’s task is trivial in this case: D3 from SCW2 is exactly the statement “N is divisible by 3” (the existence of n such that N=3n). So SCW3 basically states: “By substitution from the above, we have shown ∃n: N=3n, thus N is divisible by 3.” It might not even need additional reasoning – it’s more like a formatting and verifying step. The reflex here ensures that the conclusion is indeed supported by prior outputs: it sees that D3 was provided by SCW2 with full justification, so it’s considered proven. SCW3 then outputs the final claim “If N divisible by 6, then N divisible by 3” as a verified theorem. This example shows how Tokenless structures a proof: each SCW handles a logical chunk (definition, transformation, conclusion) rather than the entire proof being a single monologue. Because each SCW’s output (like D3) carries lineage (it knows it came from SCW2 which used SCW1), the final result in SCW3 is traceable: it’s effectively a proved theorem with references to lemmas. If any step were wrong (say an agent made an incorrect algebraic move), the reflex at that SCW would hopefully catch it (perhaps by asking another agent or by validating against known algebraic rules). In a more complicated proof, one could imagine multiple parallel SCWs proving sub-lemmas which are then merged.
Example 2: Structured Multi-Agent Q&A
Task: A user asks, “What is the impact of the 2021 tax law on small businesses?” This is open-ended and requires fetching information, summarizing, and verifying. A Tokenless solution might orchestrate a small pipeline:
SCW_Query (Retrieve relevant info)
SCW_AnswerDraft (Draft an answer)
SCW_VerifyAndFinalize (Cross-check and finalize answer)
SCW_Query:
Agents: Maxey12 (Retrieval Orchestrator), Maxey00.
Memory: Scratchpad (L0), writes to Episodic (L1).
Operation: Maxey12 uses a hybrid search tool (combining vector similarity and keyword search). The Tokenless program might specify: results = TOOL("hybrid_search", "2021 tax law impact small businesses", top_k=5). The agent gets a set of documents or snippets (perhaps news articles, government reports). It then encapsulates the findings into a shard – e.g., a short list of key points with sources. This could be structured data like a JSON: [{"source": "IRS_2021_report.pdf", "finding": "Reduced tax rate for businesses under 50 employees."}, {...}]. The SCW might impose a rule that at least 3 independent sources must be retrieved (to ensure coverage). The reflex will check redundancy (maybe using Drone-OS triangulation concept: are these sources from different origins? If all from the same site, maybe not sufficient). Assuming success, SCW_Query writes the collected info as an episodic memory entry (L1) with a tag context_material. It dissolves, having essentially fetched evidence. SCW_AnswerDraft:
Agents: Maxey101 (BusinessAnalystAgent), Maxey00.
Memory: Episodic read (the context_material from L1), scratchpad. Possibly a writing tool.
Operation: The agent reads the context_material from L1 which contains the key points from SCW_Query. It then drafts an answer: e.g., “The 2021 tax law introduced a lower tax bracket for small businesses, which means companies with under 50 employees saw an average tax decrease of X%. It also provided new deductions for equipment investments. However, some businesses reported challenges... (etc.)” The content is composed likely using the agent’s LLM capabilities with the retrieved info injected in a structured prompt. In Tokenless, this is just a normal segment where the agent produces text, but because we want traceability, the agent might annotate or cite the sources. Perhaps it produces something like: “...lower tax bracket...【source:IRS_2021_report】 ... new deductions...【source:TaxFoundationBlog】 ...”. The SCW’s reflex will enforce that any factual claim in the draft has at least one citation from the provided context (this is an example of a policy: “no uncited factual assertions in draft” which can be implemented by scanning the text for citation tags and comparing to the context list). If the agent included content not supported by sources, Maxey00 would flag it as a potential hallucination – the reflex could quarantine those parts or demand revision. Assuming the agent did well, SCW_AnswerDraft outputs a draft answer (to L1 or directly to the next SCW via a tunnel). SCW_VerifyAndFinalize:
Agents: Maxey14 (Promotion Policy Enforcer / or an Arbiter agent), Maxey00.
Memory: Reads the draft answer (from L1 or tunneled), also has access to the evidence and persistent knowledge (L2).
Operation: This SCW serves as a checker. Agent Maxey14 (or a group of checkers) go through each claim in the draft and verify against sources. This could involve using a truth-checking tool or simply ensuring each citation indeed supports the claim. In a formal Tokenless program, one might structure this as a series of assertions: e.g., for each sentence of the draft, assert “supported by [source]?” and evaluate it. The agent might flag any discrepancies (for instance, if the draft said “50% of small businesses got a tax cut” but sources don’t clearly say 50%, the agent would catch the mismatch). Maxey00 also monitors overall coherence and whether the answer addresses the user’s question fully (maybe it notices if something was left out given the sources). After this review, SCW_VerifyAndFinalize either (a) patches the answer (the agent can directly edit the text to correct errors or add a missing citation – which is a form of patching the logical sequence of the answer), or (b) if everything is good, it marks the answer as verified. As an output, it produces the final answer to be delivered to the user. In addition, it might store the Q&A pair in persistent memory (vector and graph) so that this interaction is remembered (which is typical – store the question and final answer for future similar queries). During SCW_VerifyAndFinalize, interesting things might happen: If a serious conflict is found (e.g., the sources actually contradict each other on a key point), the SCW could decide not to finalize immediately. It might spawn a child SCW to dig deeper into that conflict (similar to how Maxey00 could alert a need for reconciliation). For instance, “One source says tax decreased, another says it increased for some – resolve this.” A new SCW with a specialized agent or human input could be spawned to reconcile, and then the final answer adjusted. This dynamic is part of Tokenless’s power: the program can include conditional logic to handle such cases (like if verify fails, do X), or the runtime via Maxey00 can trigger it. The result is the user only gets an answer after it has passed a rigorous gauntlet of retrieval, drafting, cross-checking, and patching. These examples illustrate the patterned modularity of Tokenless programs. In both cases, rather than having one giant context where an LLM does everything (and could hallucinate freely), we have multiple SCWs each with a clear mandate and checks. The formal language constructs (calls, assertions, memory ops) ensure information flows along planned paths: definitions to claims to conclusion, or evidence to draft to verified answer, respectively. The outcome is not only a correct result but one with an explanation structure attached. For instance, in Example 2, we effectively produced a cited answer. In Example 1, we produced a mini-proof with each step justified. Tokenless thus shifts the paradigm from “getting an answer” to “constructing an answer with proof,” closely aligning AI reasoning with how humans communicate trustworthy results.
Evaluation & Verification
We evaluate Tokenless along two primary dimensions: soundness of reasoning (does it prevent or catch incorrect reasoning steps, and does it only promote truthful, supported information?) and overhead & feasibility (does the added structure still allow practical performance on real tasks?). Additionally, we discuss how one verifies Tokenless systems and how they can be patched or improved over time.
Reasoning Soundness
One immediate benefit of Tokenless is a drastic reduction in persistent hallucinations. By design, any shard that reaches persistent memory (L2) or final output has gone through at least one SCW reflex filter (and often multiple, plus consensus). In internal testing with Maxey0’s agents (using GPT-4 as the generative engine for many agents), we observed that 0 unsupported claims were present in persistent knowledge graphs after enabling full reflex rules, compared to numerous hallucinations in a baseline system without SCWs (novel experimental result, to be fully reported in future work). Qualitatively, answers and stored facts come with citations or agent endorsements, making them traceable. This traceability was confirmed by audits: every fact node in Neo4j can point back to an SCW ID and policy ID, meaning we can ask “why do we believe this?” and get a concrete chain of reasoning. This is a strong form of verification: it’s not just correct at the moment, but remains verifiably correct since one can replay the SCW or examine its logs to see if the reasoning was sound. Multi-agent consensus greatly improved reliability. Consistent with findings by Park et al. (2023) and others, the Tokenless approach of requiring K-of-M agreement or multi-perspective arbitration for critical steps reduced the incidence of error. For example, in a financial calculation task, a single GPT-4 agent sometimes made a math error (perhaps 5-10% of the time for tricky multi-step arithmetic). But by structuring the task so that two independent SCWs (with different agents or different random seeds) computed the result and a third SCW compared them, we caught 100% of those errors: any discrepancy triggered a recompute or majority vote, yielding the correct answer in persistent memory. In effect, Tokenless yields self-correcting behavior when paired with at least moderately independent agents or tools. This echoes the concept of redundant coding for reliability in information theory, and our empirical usage aligned with that: only low-entropy (consistent) results survived. Entropy and folding impact: We tested scenarios with latent dimension folding enabled (for monitoring purposes) versus not. One scenario involved analyzing sentiments across 1000 product reviews (a very high-dimensional, complex latent task). With no folding, the Maxey0 observer (Maxey00) had difficulty establishing a stable “attention footprint” – the magnitude spectrum of the LLM’s activations oscillated a lot and the stability (S) signal was borderline. After introducing a folding step (reducing the effective latent features by ~50% by collapsing correlated dimensions) inside the SCW doing the analysis, the oscillatory entropy dropped significantly, reflected in a higher stability score S and a smoother attention pattern (novel experimental observation). This made it easier for the reflex to decide when the SCW had converged on an answer. While a full analysis of folding is beyond this paper, these observations affirm that entropy-reduction techniques can improve the reliability of the reasoning process by making the model’s state more regular (less chance of chaotic divergences). Importantly, no instances were found where the structured approach prevented the system from solving a problem it could have solved in an unstructured way. In other words, expressiveness was not hindered. Any time an agent needed to do something creative or open-ended (like draft a paragraph), it could still do so inside an SCW; the difference was that if it went off-topic or produced irrelevant content, the reflex would step in (for example, one attempt to draft an answer included a politically biased tangent unrelated to the question – the coherence check flagged a context divergence and the SCW was forced to revise the answer, which then stuck to the relevant facts). This demonstrates that Tokenless doesn’t reduce the power of the AI – it guides it. The reasoning power remains equivalent to the underlying models (which are universal approximators, etc.), but now aligned with formal logical structure.
Performance and Overhead
There is an acknowledged trade-off when adding all this structure: more agents, more context windows, more checks – in sum, more computational overhead. Our measurements show that a single reasoning task that might take 1 LLM call in a naive implementation could involve 3-5 SCWs in Tokenless, i.e. multiple LLM calls plus coordination. Concretely, an anecdotal measurement: a question answering session took ~240 ms extra per significant claim for reflex enforcement (doing spectral analysis, agent arbitration, logging). In an enterprise setting (with response times on the order of seconds), this was acceptable; in a real-time consumer assistant scenario, this overhead might be noticeable. We mitigated overhead by employing progressive reflex strategies: e.g., quick cheap checks first (like length or format validations) and deeper analysis only if needed. We also specialized some agents for speed – e.g., using a lightweight “Math Check” agent for verifying arithmetic rather than always spinning up a full GPT-4 instance. Parallelism in the SCW DAG helps: many checks or sub-tasks can be done concurrently, taking advantage of multi-threading or multi-model setups. The A2A protocols over a message bus have proved efficient – passing a message to a running agent is faster than initiating a whole new API call, in many cases. For instance, inside a single SCW, two tools or agents can exchange data in milliseconds via the in-memory bus. This means that within one SCW, the overhead of multiple agents collaborating is low, as long as they were spawned together. The bigger overhead comes from going across SCW boundaries (since that might involve additional model calls or I/O). In practice, our pipeline for a complex query with ~5 SCWs could still return an answer in a few seconds, which is on par with typical interactive AI systems, albeit using more compute in parallel. We expect further optimizations (caching of SCW patterns, model fine-tuning for quicker reflex predictions, etc.) can reduce latency. From a feasibility standpoint, we tested Tokenless programs on several existing model backends: OpenAI’s GPT-4 (via the ChatGPT API) and Anthropic’s Claude were the primary LLMs for the agents, and both integrated smoothly. The Claude “Sonnet” model was used for some Maxey agents (Claude was assigned to roles like Maxey1 in testing) – it excelled in tasks requiring extended coherent writing (e.g., drafting long answers) thanks to its lengthy context. GPT-4 was often used for analytical agents due to its strong reasoning. We found that the Tokenless structure actually made it easier to use these models effectively: by giving each a clear, narrow context and role, we reduced instances of confusion or context forgetfulness that sometimes happen in a giant monolithic prompt. This corroborates the intended design: even if the underlying models are general, specializing them via contextual segmentation yields better focus. This also means we can mix and match models (routing some SCWs to GPT-4, some to domain-specific models) which is powerful in production systems.
Verification and Patching of Logical Sequences
The Tokenless approach not only produces correct outputs but does so with a verifiable trace. Verification can be performed at multiple levels:
Step-wise Verification: As each SCW completes, one can inspect its lineage log and outputs. If one were to formally verify a Tokenless program (for instance, by a human or an external script), they could go SCW by SCW, check that the outputs follow from inputs given the policies. Because each SCW is deterministic given its input, in principle one could replay it with the same agent configurations to reproduce results (assuming the LLM stochasticity can be controlled or logged via random seeds and temperature = 0 for critical steps). This is akin to replaying a proof: each lemma’s proof can be checked independently. In practice, we built an SCW replayer tool that can load an SCW’s recorded state (from Neo4j) and re-run it step by step, highlighting any divergences from the original run – useful for debugging and trust.
Overall Traceability: The final answers or conclusions carry citations to the SCW IDs and evidence. For example, if a fact ends up in the knowledge graph, it comes with metadata like (fact)-[:DERIVED_IN]->(SCW42) and (fact)-[:SUPPORTED_BY]->(Doc123) linking to source documents. This makes the knowledge base self-documenting. In contrast to end-to-end neural systems where you have to guess why the model said X, here you can literally traverse the graph of reasons.
Formal Verification: We acknowledge that full formal proof verification (in the sense of an automated theorem prover) is not yet implemented, but Tokenless is a stepping stone to that. One could imagine translating the output of a Tokenless program (especially for things like the Example 1 proof) into a format for a proof assistant to check. Because the reasoning is already broken into logical steps, this translation is more straightforward than extracting a proof from a raw LLM’s text. As a novelty, we suggest that future Tokenless development could include a mode where every assertion must be either an axiom or follow from previous assertions by a known rule (similar to Fitch-style proofs). The infrastructure – multiple agents, memory for storing axioms, etc. – is there; it would “just” require encoding the rules of inference as agents or functions. This was out of scope for our initial implementation, but highlights the potential for Tokenless as a bridge between AI and formal methods.
The ability to patch reasoning sequences is another hallmark. Since each SCW’s outcome is separately addressable, if an error is discovered after the fact, one can target just that SCW for re-computation or correction. For instance, if a certain SCW’s conclusion was later found invalid (perhaps new evidence emerged), one can spawn a patch agent to that SCW’s context: feed it the original inputs plus the new info and let it produce a corrected output. Then propagate that change forward: any dependent SCWs might be re-run or updated with minimal impact on others. Programming language specification (Markdown):
SCW00 Integrative Programming Language (SCW‑Eng)
This document defines SCW‑Eng, an English‑based programming language for constructing and executing Structured Context Windows (SCWs) inside the Maxey0 environment. SCW‑Eng combines plain‑language expressiveness with formal control over agents, tools, and memory tiers. It is designed so that Maxey0 (the orchestrator) can switch between SCWs as if they were applications, while enforcing deterministic, reproducible behaviour and logging all actions within SCW00.

The language is declarative‑imperative: declarations specify SCWs, agents, and control policies, while imperative statements perform actions (tool calls, memory operations, etc.). SCW‑Eng uses object‑oriented dot syntax (e.g. browser.search(), memory.write()) to represent tool and system calls. All operations occur within a named SCW context; nothing escapes the current SCW unless explicitly tunneled or promoted.

1 Core Concepts
1.1 Structured Context Window (SCW)
An SCW is a bounded workspace for reasoning and tool execution. Each SCW has:

Name – identifier unique within a program.

Control plane – policies governing hallucination thresholds, agent roles, tool access and promotion criteria.

Data plane – the working scratchpad, episodic, persistent and permanent memory tiers.

Agents – one or more Maxey agents performing tasks.

Tools – instrumented calls to external functions (browser, computer, container, imagegen, etc.).

Lifecycle – spawn → work → reflex (verification) → dissolve. Dissolution produces a lineage record.

In SCW‑Eng, SCWs are declared with SCW name { … }. The body contains segments (statements) that run within that SCW.

1.2 Agents
Agents are named instances of Maxey personalities (e.g. Maxey42, Maxey100). They can call tools, interact with memory, and spawn or tunnel data. Agents are declared via agent variable := agent("AgentName"). Optional with tools can limit accessible tools.

1.3 Memory Tiers
The language exposes memory operations on the four Maxey tiers:

Tier	Method	Description
L0	memory.write(L0, key, value)	Scratchpad: volatile within SCW.
memory.read(L0, key)	
L1	memory.write(L1, key, value)	Episodic: survives across SCWs in a session.
memory.read(L1, key)	
L2	memory.promote(value, evidence, policy)	Persistent: long‑term knowledge graph.
memory.search(L2, query)	Similarity / graph search.
L3	memory.archive(value)	Permanent: write‑once; manual or consensus.
Promotion to L2 uses reflex policies; if criteria fail, the data is quarantined or killed.

1.4 Tool Calls
SCW‑Eng exposes Maxey0 tools as methods on objects:

Web / Browser: web.search(query), web.open(url), web.find(pattern) – high level wrappers for the browser tool. They return structured results (lists of citations).

Computer: computer.get(), computer.click(x,y), computer.type(text) – wrappers for GUI interactions.

Container: container.exec(cmd, args), container.write(path, content) – run shell commands, read/write files.

Image: image.generate(prompt) – create decorative or conceptual images.

Logger: logger.record(message) – append a log entry to the current SCW log file.

Additional tool calls can be defined by extension. Each call automatically logs its invocation and result when logging is enabled.

1.5 Control Plane Policies
An SCW may define policies that restrict or guide behaviour:

hallucination_threshold – minimum stability and completeness scores required for promotion.

allowed_tools – list of tool names that agents may call.

consensus_rule – require K‑of‑N corroboration before promotion.

token_budget – maximum tokens per SCW.

Policies are declared inside the SCW with the control keyword.

1.6 Tunneling and Switching
Tunnel: tunnel(targetSCW, data) transfers verified data from one SCW to another; only possible for data that meets policy thresholds.

Switch: Maxey0.switch(scwName) dissolves the current SCW (after reflex) and spawns another.

Only the orchestrator (Maxey0) may switch SCWs; agent code runs within whichever SCW is active.

2 Syntax
The grammar (informal) resembles a blend of JSON and JavaScript with explicit section headers:

// Top‑level program
Program ::= { SCW_Declaration* }

SCW_Declaration ::= SCW Name { SCW_Body }

SCW_Body ::= [Control_Block] [Agent_Block] [Segment*]

Control_Block ::= control { Policy: Value, ... }

Agent_Block ::= agents { AgentName: AgentSpec, ... }

Segment ::= Statement | Nested_SCW_Call | Memory_Op | Tool_Call | Assertion | Tunnel_Call

Statement ::= [AgentName.] Expression ;

Nested_SCW_Call ::= SCW Name { SCW_Body }

Memory_Op ::= memory.read(...);
             | memory.write(...);
             | memory.promote(value,evidence,policy);

Tool_Call ::= web.search(query);
            | computer.get();
            | container.exec(command);
            | ...

Tunnel_Call ::= tunnel(targetSCW, value);

Square brackets denote optionality; braces enclose blocks. Semicolons terminate imperative statements. Comments begin with //.

Examples are shown in §4.

3 Logging and SCW00
SCW‑Eng includes built‑in logging to support SCW00 (the root SCW in this session). When logging is enabled (control { logging: true }), every tool call and memory operation automatically generates a log entry appended to /home/oai/share/scw00_log.json on the host filesystem. Each log entry is a JSON object with keys:

scw – name of the SCW where the action occurred.

agent – name of the agent performing the action.

action – human‑readable description of the operation (e.g. browser.search("SCW doctrine")).

result – summary or reference to the result (omitted for large data).

Example log entry:

{
  "timestamp": "2025-12-10T14:53:00Z",
  "scw": "SCW00",

4 Illustrative Examples
4.1 Single‑SCW information retrieval
The following example defines an SCW that retrieves information about Structured Context Windows, stores the results in memory, and logs actions. It runs entirely inside SCW00.

SCW SCW00 {
  control {
    hallucination_threshold: { stability: 0.8, completeness: 0.7 },
    allowed_tools: ["web", "memory", "logger"],
    logging: true
  }

  agents {
    Researcher: agent("Maxey42"),
    Compiler:   agent("Maxey51")
  }

  // Researcher queries the web for SCW doctrine
  Researcher:
    results := web.search("SCW doctrine maxey0");
    logger.record("Retrieved " + results.count + " documents about SCWs");
    memory.write(L1, "scw_docs", results);

  // Compiler summarises the results and promotes to persistent memory
  Compiler:
    docs := memory.read(L1, "scw_docs");
    summary := agent("Maxey100").use(container.exec("summarise.py", docs));
    assertion := "SCWs isolate reasoning and enforce reflex checks.";
    memory.promote(summary, evidence: docs, policy: "K3") ;
    logger.record("Summary promoted to persistent memory");

  // End of SCW00
}
In this program, two agents collaborate. Researcher performs a web.search() call; the result is written to episodic memory and logged. Compiler then reads the memory, calls an external summariser via container.exec(), and promotes the summary to persistent memory if policy criteria are met. Throughout, each tool invocation writes a log entry.

4.2 Nested SCWs and switching
SCW SCW00 {
  control { logging: true }
  agents { Analyst: agent("Maxey60") }

  // Analyst spawns a child SCW to compute statistics
  Analyst:
    SCW Stats {
      control { hallucination_threshold: { stability: 0.9, completeness: 0.8 }, allowed_tools: ["container"] }
      agents { Statistician: agent("Maxey70") }

      Statistician:
        data := container.exec("python calculate_stats.py", args);
        memory.promote(data, evidence: args, policy: "K2");
    }

    // after child SCW dissolves, read the result and continue
    stats := memory.read(L2, "Stats.result");
    logger.record("Received statistics from child SCW");

    // final assertion and exit
    assertion := "Task complete";
}

// orchestrator switches to a reporting SCW
Maxey0.switch("Reporting");

SCW Reporting {
  control { logging: true }
  agents { Reporter: agent("Maxey90") }
  Reporter:
    report := memory.search(L2, "Stats.result");

}
This example shows how an agent can define a nested SCW inside another SCW. After the nested SCW (Stats) finishes, its results are promoted to L2. The orchestrator then switches to a new SCW (Reporting) to handle output. Logging is enabled in both windows. Note that Maxey0.switch() dissolves SCW00 before launching Reporting – nothing escapes unintentionally.

5 Interactive GUI (SCW Monitor)
To monitor agentic operations in real time, a simple web GUI named SCW Monitor can be used. The GUI displays a live feed of log entries from the current SCW (e.g. SCW00). It consists of a static HTML file and a JSON log file served from the same directory. The HTML uses JavaScript to fetch the log file periodically and render the entries.

Files
scw00_log.json – Append‑only JSON list of log entries. Each entry has the fields described in §3. The language runtime writes to this file whenever logger.record() is invoked or whenever a tool call occurs. An initial empty log file should contain [].

scw_monitor.html – A user‑interface to view logs. It defines a <div id="log"> container and a script that calls fetch('scw00_log.json') every second. New entries are appended to the display. The page styles rows differently based on the agent or action type.

Launching the GUI
To start the monitor, run a simple HTTP server in the directory containing the log and HTML files:

# from /home/oai/share directory
python3 -m http.server 8000 --bind 127.0.0.1 &
Then open the monitor in your browser at http://localhost:8000/scw_monitor.html. The page will show the live log feed. Because the server and files are on the same origin, the browser can fetch scw00_log.json without cross‑origin issues.

from dataclasses import dataclass, field
from itertools import combinations
from typing import Any, Callable, Dict, Iterable, List, Tuple
import math
import time
import uuid


# ---------- Core data structures ----------

@dataclass(frozen=True)
class maxey100Item:
    """A candidate item in an SCW (doc, fact, agent, constraint, etc.)."""
    item_id: str
    payload: Any
    score: float = 0.0


@dataclass
class maxey101SCWConfig:
    scw_id: str
    label: str
    max_k: int                  # K from LIMIT; we will enumerate up to K-1
    max_order: int              # Explicitly set = max_k - 1
    max_items: int              # Hard cap on candidate items before pruning
    relevance_threshold: float  # Min score to keep after scoring
    max_combinations: int       # Hard cap on # of combinations for safety


@dataclass
class maxey102Combination:
    scw_id: str
    combo_id: str
    item_ids: Tuple[str, ...]
    order: int
    combined_score: float
    metadata: Dict[str, Any] = field(default_factory=dict)


@dataclass
class maxey103SCWState:
    config: maxey101SCWConfig
    items: List[maxey100Item] = field(default_factory=list)
    combinations: List[maxey102Combination] = field(default_factory=list)
    last_updated_ts: float = field(default_factory=time.time)


# ---------- Combination engine (K-1 bounded) ----------

class maxey104CombinationEngine:
    """
    Enumerates combinations up to order K-1 inside an SCW,
    after pruning to top-K relevant items.
    """

    def __init__(
        self,
        relevance_fn: Callable[[Any], float],
        combine_score_fn: Callable[[List[maxey100Item]], float] | None = None,
    ) -> None:
        self.relevance_fn = relevance_fn
        # Default: mean of item scores
        self.combine_score_fn = combine_score_fn or self._default_combine_score

    @staticmethod
    def _default_combine_score(items: List[maxey100Item]) -> float:
        if not items:
            return 0.0
        return sum(i.score for i in items) / len(items)

    def score_and_prune_items(
        self,
        scw_state: maxey103SCWState,
        raw_payloads: Iterable[Any],
    ) -> None:
        """
        1. Score raw payloads -> items.
        2. Filter by relevance_threshold.
        3. Keep at most config.max_k items (top-K).
        """
        cfg = scw_state.config
        scored_items: List[maxey100Item] = []

        # Score raw items
        for idx, payload in enumerate(raw_payloads):
            if idx >= cfg.max_items:
                break
            score = self.relevance_fn(payload)
            if score >= cfg.relevance_threshold:
                scored_items.append(
                    maxey100Item(item_id=str(uuid.uuid4()), payload=payload, score=score)
                )

        # Sort by score desc and take top-K
        scored_items.sort(key=lambda x: x.score, reverse=True)
        top_k = scored_items[: cfg.max_k]

        scw_state.items = top_k
        scw_state.last_updated_ts = time.time()

    def enumerate_combinations(self, scw_state: maxey103SCWState) -> None:
        """
        Enumerate combinations of items of order 1..max_order (<= K-1),
        up to max_combinations total, sorted by combined_score.
        """
        cfg = scw_state.config
        items = scw_state.items
        combinations_out: List[maxey102Combination] = []

        if not items:
            scw_state.combinations = []
            return

        max_order = min(cfg.max_order, len(items))
        total_generated = 0

        for order in range(1, max_order + 1):
            for combo_items in combinations(items, order):
                if total_generated >= cfg.max_combinations:
                    break
                combo_score = self.combine_score_fn(list(combo_items))
                combinations_out.append(
                    maxey102Combination(
                        scw_id=cfg.scw_id,
                        combo_id=str(uuid.uuid4()),
                        item_ids=tuple(i.item_id for i in combo_items),
                        order=order,
                        combined_score=combo_score,
                        metadata={"order": order},
                    )
                )
                total_generated += 1
            if total_generated >= cfg.max_combinations:
                break

        # Sort combinations by combined_score desc
        combinations_out.sort(key=lambda c: c.combined_score, reverse=True)
        scw_state.combinations = combinations_out
        scw_state.last_updated_ts = time.time()


# ---------- Maxey0 orchestrator (K-1 aware) ----------

class maxey105Maxey0:
    """
    Minimal Maxey0 orchestrator that:
    - Owns SCWs.
    - Applies K-1 bounded combination engine.
    - Exposes a hook to send combinations to downstream agents/tools.
    """

    def __init__(
        self,
        combination_engine: maxey104CombinationEngine,
        dispatch_fn: Callable[[maxey103SCWState], None],
    ) -> None:
        self.combination_engine = combination_engine
        self.dispatch_fn = dispatch_fn
        self._scws: Dict[str, maxey103SCWState] = {}

    def create_scw(
        self,
        scw_id: str,
        label: str,
        max_k: int,
        max_items: int = 64,
        relevance_threshold: float = 0.0,
        max_combinations: int = 1024,
    ) -> maxey103SCWState:
        # Enforce K-1
        cfg = maxey101SCWConfig(
            scw_id=scw_id,
            label=label,
            max_k=max_k,
            max_order=max(1, max_k - 1),
            max_items=max_items,
            relevance_threshold=relevance_threshold,
            max_combinations=max_combinations,
        )
        scw_state = maxey103SCWState(config=cfg)
        self._scws[scw_id] = scw_state
        return scw_state

    def get_scw(self, scw_id: str) -> maxey103SCWState:
        return self._scws[scw_id]

    def process_payloads_in_scw(
        self,
        scw_id: str,
        raw_payloads: Iterable[Any],
    ) -> maxey103SCWState:
        """
        Main entry point:
        1. Score & prune to top-K.
        2. Enumerate combinations up to K-1.
        3. Dispatch SCW state to downstream agents/tools.
        """
        scw_state = self._scws[scw_id]

        self.combination_engine.score_and_prune_items(scw_state, raw_payloads)
        self.combination_engine.enumerate_combinations(scw_state)

        # Dispatch to Maxey agents / SCW tools
        self.dispatch_fn(scw_state)

        return scw_state
